Dans cette vidéo, nous allons parler du protocole de contrôle de transmission, TCP. Si vous n'avez pas regardé la vidéo sur le protocole internet, IP, vous voudrez peut-être le faire avant de regarder cette vidéo, car les deux sont assez inter-reliés. Ainsi, le protocole internet, encore une fois, en résumé rapide, c'est le protocole qui déplace les informations d'une machine émettrice à une machine réceptrice via le réseau.

Alors, qu'est-ce que TCP ? Juste déplacer d'une machine émettrice à une machine réceptrice ne raconte pas toute l'histoire. Nous savons également que notre programme, nos ordinateurs, par exemple, exécutent plusieurs programmes et ont plusieurs services en cours d'exécution sur ces machines. Et donc, si nous voulons obtenir un paquet ou des informations vers un programme spécifique sur une machine spécifique, nous avons besoin de plus d'informations que ce qu'IP nous permet d'obtenir pour obtenir des informations de A à B.

TCP peut donc être considéré comme dirigeant le paquet vers le programme ou le service correct sur la machine réceptrice. Et donc, il est important de, comme vous pourriez vous y attendre, savoir où il est censé aller et à quoi sert le paquet en même temps. Et donc, fréquemment, lorsque vous parlez du protocole de contrôle de transmission, TCP, vous entendez souvent cela dans le contexte, TCP/IP ou juste TCP/IP. Ces deux protocoles sont si inter-reliés qu'ils sont essentiellement traités comme une seule unité. Mais ce sont deux protocoles distincts qui font deux choses distinctes.

Encore une fois, IP est responsable de l'obtention des informations d'une machine à une autre. Et TCP est responsable de faire parvenir des informations au programme ou au service correct sur une machine. Et il fait quelque chose d'autre que IP ne fait pas, qui est de garantir la livraison.

Ainsi, si nous associons maintenant l'adresse IP d'une machine au numéro de port, et un numéro de port est la façon dont un service spécifique ou un programme est identifié sur une machine. Si nous avons maintenant une adresse IP plus un numéro de port, nous pouvons maintenant identifier de manière unique un service particulier en cours d'exécution sur une machine particulière.

C'est pourquoi TCP et IP sont si fréquemment inter-reliés, car ce numéro de port à lui seul ne signifie pas grand-chose si vous avez besoin d'un numéro de port et de la machine dont vous parlez. Quelle machine doit utiliser ce port particulier, par exemple.

L'autre chose que TCP fait, comme je l'ai dit, est de garantir la livraison. Ainsi, en plus de spécifier le numéro de port, il indique également combien de paquets le protocole internet, IP, a divisé les données. Et il ordonne ces paquets pour qu'ils puissent être reconstitués sur la machine réceptrice, même s'ils ont été reçus dans un ordre différent de celui dans lequel ils ont été envoyés. Ce qui peut arriver car IP est un protocole sans connexion, et donc différents paquets peuvent prendre des chemins différents à travers le système.

Certains de ces numéros de port sont très couramment utilisés et ont été standardisés sur tous les ordinateurs, comme, à peu près tous les fabricants d'ordinateurs maintenant. Ainsi, quelque chose appelé FTP, le protocole de transfert de fichiers, qui est utilisé pour transmettre des fichiers, comme vous pouvez vous y attendre, d'une machine à une autre, utilise conventionnellement le port 21. L'e-mail, SMTP, utilise le port 25. Le DNS, le système de noms de domaine, dont nous avons parlé dans notre vidéo internet d'introduction, utilise le port 53. Si vous naviguez sur le web, vous utilisez presque toujours le port 80, sauf si vous naviguez sur le web en toute sécurité en utilisant le port 443.

Alors, quel est ce processus TCP/IP ? Que se passe-t-il avec ces deux protocoles ensemble ? Eh bien, parlons-en. Lorsqu'un programme souhaite envoyer des données, TCP aide à les diviser en morceaux et communique ces paquets au logiciel en réseau de l'ordinateur. Ainsi, il prend les données et l'enveloppe d'informations qui indiquent à quel port elle est censée aller et dans quel ordre ce paquet est parmi tous les autres. Donc, il fait le paquet un sur dix, deux sur dix, trois sur dix, et ainsi de suite.
IP reçoit ces morceaux de données enveloppés avec TCP et enrobe plus d'informations sur où le paquet doit aller. Nous pourrions appeler cela les couches IP entourant le paquet. Donc, c'est un peu comme une poupée emboîtée. Nous avons les données au milieu, puis TCP par-dessus, indiquant où les données à l'intérieur de TCP sont censées aller, vers quel port ou quel service sur une machine. Autour de cela se trouve la couche IP. Quelle adresse IP, quelle machine reçoit réellement ceci.

Ensuite, ce paquet qui a été enveloppé avec toutes ces couches est envoyé à travers le protocole internet via le système de routeurs, de A à B. Lorsque la machine ou le dispositif récepteur le reçoit, il regarde la couche IP, il dit, ouais, c'est mon adresse IP, alors il enlève, en quelque sorte, la coque et enlève la couche IP. Puis il voit qu'il y a une couche TCP et il dit, OK, on dirait que cela va aller au port X ou au port Y. Et apparemment, c'est le paquet huit sur 15. Donc, c'est bon à savoir. Ensuite, il peut prendre ces informations, retirer la couche TCP maintenant, sachant que c'est pour le port X et le paquet huit, et accéder aux données à l'intérieur. Et il peut préparer les données à être organisées de la manière correcte. Et une fois que l'ensemble des données est reçu, TCP peut le remettre au service correct et dire : "Le voici. Voici les données que vous avez reçues". Ce processus pourrait ressembler à ceci. Ainsi, envoyons un e-mail d'un expéditeur à un destinataire.
